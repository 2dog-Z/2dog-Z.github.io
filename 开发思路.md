# 开发思路（静态终端风格个人主页）

## 1. 项目定位与目标

本项目是一个纯静态的个人主页 Demo：上半区左侧展示内容、右侧展示评论，下半区提供一个“仿终端”的交互区。终端命令可以在一个内置的“虚拟文件系统”中浏览目录、读取文章，并能通过 `say` 将消息追加到右侧评论列表。

核心目标：

- **零依赖、可直接打开**：无需构建与安装，浏览器打开 `index.html` 即可运行。
- **终端式交互体验**：用最少的状态与 DOM 操作实现命令行浏览/阅读。
- **结构清晰、易扩展**：增加文章/目录/命令时，尽量只改一个位置或一小段逻辑。

## 2. 技术选型与约束

- **HTML + CSS + 原生 JavaScript**
- **无后端、无路由、无打包器**：所有数据与“文件系统内容”内置在 `app.js` 中。
- **渲染策略**：完全基于 DOM API（`createElement` / `appendChild` / `textContent`），避免引入模板与框架的复杂度。

## 3. 目录结构（最小可运行集）

```
Version 1/
  ├─ index.html   # 页面骨架：内容区/评论区/终端区
  ├─ styles.css   # 整体布局与终端/评论样式
  └─ app.js       # 交互逻辑：虚拟文件系统、终端命令、评论渲染
```

## 4. 页面结构设计（从布局出发）

页面分为两大块：

1. **上半区 `.top`**
   - 左侧 `.content`：标题与说明文字 + 命令速查表
   - 右侧 `.comments`：评论列表（仅前端内存态）
2. **下半区 `.terminal`**
   - 输出区 `#terminalOutput`
   - 输入区：提示符 `#promptText` + 输入框 `#terminalInput` + 光标动画

对应文件：

- 页面骨架：`index.html`
- 布局与主题：`styles.css`（通过 CSS 变量实现“VS Code dark”风格调色板）

## 5. JavaScript 架构：数据 + 模块 + 状态

整体设计分三层：**静态数据** → **工具函数** → **两个功能模块（评论、终端）**。

### 5.1 静态数据

1. **命令速查表**
   - 用于 `help` 输出时生成固定格式文本，也能作为“可用命令的单一真源”之一。
2. **虚拟文件系统（FILE_SYSTEM）**
   - 用一个嵌套对象模拟目录结构：
     - 目录：对象 `{ ... }`
     - 文件：字符串（文件内容）
   - 好处：`ls/cd/cat` 的实现可以复用同一套路径解析逻辑。

### 5.2 工具函数

工具函数聚焦于“终端交互最小必需能力”：

- `randomNickname()`：生成随机昵称（用于初始化评论与新增评论）
- `formatDate(d)`：格式化日期为 `YYYY-MM-DD`
- `tokenize(input)`：把输入拆成 `{ cmd, args }`
- `joinPath(parts)`：把当前路径数组渲染成 `~/a/b`
- `resolveDir(fs, pathParts)`：从虚拟文件系统解析某个目录节点
- `listDir(fs, pathParts)`：列出目录内容（区分目录/文件并排序）
- `treeAtRoot(fs)` / `treeAtPath(fs, pathParts)`：生成树状展示文本（用于 `ls` 和 `cd` 反馈）

这些函数的目标是：

- **单一职责**：每个函数只做一件小事，方便替换/扩展。
- **纯函数优先**：除了 DOM 渲染相关函数，多数工具函数不依赖外部状态，便于排查问题。

### 5.3 评论模块（右侧面板）

评论模块的职责：

- 初始化一组示例评论（展示效果）
- 提供 `add(text)` 方法：把新的评论插入到列表顶部

设计要点：

- 评论渲染采用 `createCommentElement()` 生成 DOM，避免字符串拼接造成的 XSS 风险（使用 `textContent`）。
- `say` 命令只负责调用 `window.__comments.add(...)`，实现“命令系统”和“评论系统”的低耦合。

### 5.4 终端模块（下方面板）

终端模块 `createTerminal()` 是交互核心，内部维护最关键的状态：

- `cwd: string[]`：当前虚拟路径（数组形式便于 `cd ..`、拼接等）
- `isPrinting: boolean`：打字机输出期间禁止输入，避免输出与输入交错
- `history: string[]` 与 `historyIndex`：上下键历史命令

终端模块提供三类能力：

1. **输出能力**
   - `appendLine(text, variant)`：立即输出一行
   - `typewriter(text, options)`：逐字打印（更强的“终端”氛围）
2. **命令处理能力**
   - `handleCommand(raw)`：统一入口，先记录历史并回显，再分发给命令实现
3. **输入事件能力**
   - `onKeyDown(e)`：Enter 执行、Up/Down 切历史

## 6. 关键业务流程（从用户操作到页面变化）

### 6.1 终端输入执行流程

用户在输入框输入并按下 Enter：

1. `onKeyDown` 捕获 Enter
2. 读取输入 → 清空输入框
3. 调用 `handleCommand(raw)`
4. `tokenize` 拆出 `cmd/args`
5. 回显一行 `guest@home:~/...$ <raw>`
6. 分发到对应命令：
   - `help` → 输出所有命令
   - `clear` → 清空输出区
   - `ls` → 输出当前路径的目录树/列表
   - `cd` / `..` → 更新 `cwd` 并输出树状结构
   - `cat` → 在终端输出文件内容
   - `say` → 追加到评论区 + 输出确认信息
   - 其他 → 输出 Unknown command

### 6.2 评论追加流程（`say`）

1. `say <text>` 进入 `runSay`
2. 若为空：提示用户输入消息
3. 调用 `window.__comments.add(text)`
4. `add` 生成 comment DOM，并插入列表首位

说明：评论目前是**内存态**，刷新即丢失，这是“纯静态、零依赖”的取舍结果。

## 7. 可扩展点（保持现有结构的前提下）

### 7.1 增加文章/目录

目标：让 `ls/cd/cat` 能看到新的内容。

做法：直接在 `FILE_SYSTEM` 中新增节点即可。

- 新目录：添加一个对象字段
- 新文件：添加一个字符串字段（内容支持多行，建议使用 `[].join("\n")`）

这条路径是最“数据驱动”的扩展方式，几乎不需要改动命令逻辑。

### 7.2 增加新命令

推荐步骤（保持一致性）：

1. 在命令速查表中增加一条（用于 `help` 输出格式统一）
2. 在 `handleCommand` 中增加分发分支
3. 为命令实现单独写一个 `runXxx` 函数，避免 `handleCommand` 变大
4. 如果命令需要访问虚拟文件系统，复用 `resolveDir/listDir/treeAtPath`

### 7.3 内容区联动（可选的下一步演进）

当前 `cat` 是在“终端输出区”展示文件内容。如果希望把文章显示到左侧 `.contentBody`，可以考虑：

- 让 `cat` 除了输出终端内容外，同时把文件内容渲染到 `.contentBody`
- 增加一个“当前打开文件”的状态（例如 `currentDoc`）
- 为内容区增加一个简单的渲染函数（保持 `textContent`，避免注入风险）

### 7.4 评论持久化（可选）

在不引入后端的情况下，最轻量的方案是 `localStorage`：

- 初始化时读取本地存储，渲染已有评论
- `add` 时同步写回存储
- 可再增加一个 `clear-comments` 命令重置

## 8. 质量与可维护性要点

- **安全**：所有用户输入只进入 `textContent`，避免 XSS。
- **可访问性**：终端输出区使用 `aria-live="polite"`；评论区设置了 `role="list"`/`listitem`。
- **交互一致性**：打字机输出期间禁用输入处理，避免输出与输入混乱。
- **性能**：输出区可能持续增长，后续可增加“最大行数”策略（达到上限时移除最旧节点）。

## 9. 本地运行与验证清单

运行方式（任选其一）：

- 直接用浏览器打开 `index.html`
- 使用任意静态服务器（例如 VS Code Live Server）打开项目目录

验证清单：

- 页面布局：上半区左右分栏、下半区终端固定可用
- 终端命令：`help/ls/cd/../cat/say/clear` 行为符合预期
- 历史命令：Up/Down 可切换输入历史
- 评论区：`say` 后右侧顶部出现新增评论
