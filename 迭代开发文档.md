# 迭代开发文档：Version 1 → Version 3

本文档用于记录本静态终端风格个人主页从 Version 1 演进到 Version 3 的核心变化、设计取舍与可扩展方式，并给出面向后续版本的迭代建议。

## 1. 项目定位与约束

项目目标是一句话：**用纯静态站点实现“像终端一样浏览内容”的个人主页体验**。

共同约束（V1~V3 一直遵守）：

- 无后端、无数据库、无打包器（浏览器可直接运行）
- HTML + CSS + 原生 JavaScript
- 交互核心在下方终端：通过命令驱动内容浏览与轻量互动（评论）

在约束下的“演进方向”：

- 从“单页内置内容”逐步过渡到“多页面内容可加载”
- 从“单文件堆叠功能”逐步过渡到“可维护的模块边界”
- 从“可用即可”逐步过渡到“可发现、可扩展、可复用”

## 2. 版本总览

### 2.1 Version 1：MVP（单文件、内容内置）

版本目标：快速做出可交互的终端主页雏形，验证布局与交互是否成立。

关键特性：

- 页面三分区：内容区 / 评论区 / 终端区
- 终端命令：`help`、`ls`、`cd`、`..`、`cat`、`say`、`clear`
- “虚拟文件系统”内置在 JS 对象里，文件内容以字符串形式硬编码
- `cat` 直接在终端输出文件内容
- 可直接 `file://` 打开 `index.html` 运行（无网络请求）

### 2.3 Version 3：模块化重构（装配式入口 + 交互增强）

关键变化（相对 V2）：

- 代码模块化拆分：`modules/` 下按职责分层，入口只做装配与启动顺序

- 主题切换与持久化逻辑独立封装，终端通过依赖注入调用
- 内容区渲染加入切换动画与失败提示，更贴近“单页体验”

## 3. Version 1 vs Version 3：对比总表

| 维度 | Version 1 | Version 3 |
|---|---|---|
| 内容来源 | JS 内置字符串（虚拟文件内容） | 真实静态页面（HTML 文件），运行时加载并注入 |
| `cat` 行为 | 在终端输出“文件内容” | 驱动左侧内容区打开页面，并在终端反馈 opened/failed |
| 虚拟文件系统 | 目录 → 文件内容字符串 | 目录 → 文件 URL/相对路径（更贴近真实站点） |
| 页面结构 | 内容区固定文案 + 静态命令表格 | 内容区是“viewport”，默认渲染 `DEFAULT_PAGE` |
| JS 组织 | 单文件 `app.js`，职责混杂 | `modules/` 模块化 + 入口装配，职责边界清晰 |
| 终端交互 | 纯文本输出 | 可点击跳转（`ls` 输出可点）、支持绝对路径、支持省略扩展名、彩蛋锁定等 |
| 主题 | 单一暗色 | `dark/light` 切换，样式表动态切换并持久化 |
| 运行方式 | 可直接 `file://` 打开 | 内容加载依赖 `fetch`，建议静态服务器（`file://` 会受限制） |
| 扩展方式 | 改 `FILE_SYSTEM` 及内容字符串 | 增加 HTML 页面 + 更新 `FILE_SYSTEM` 路由映射 |

## 5. 关键设计决策（为什么 V3 要这么改）

### 5.1 从“内置内容”到“真实页面加载”

V1 的内容模型是“文件即字符串”，优点是：

- 完全离线、无需网络请求、可直接双击打开
- `cat` 逻辑极简（读取字符串即可输出）

但当内容变多时，问题也会迅速变明显：

- JS 文件膨胀：文章越多，`app.js` 越难维护与审阅
- 内容与交互强耦合：改一篇文章也会动到核心交互代码
- 无法复用“页面级结构”：例如文章页想要自己的 HTML 结构、列表、链接等

因此 V2/V3 改为：虚拟文件系统只保存“内容地址”，内容区通过 `fetch + DOMParser` 注入，实现“静态多页面 + 单页体验”的折中。

### 5.2 模块化与依赖注入（让入口更像装配线）

V3 将职责拆分成独立模块，并在入口统一组装：

- 终端只负责交互与命令路由，通过注入的 `renderPath/applyTheme/readTheme` 调用外部能力
- 评论/主题/内容渲染各自封装 DOM 与状态逻辑
- 常量与虚拟文件系统集中定义，避免硬编码分散

这种结构的收益：

- 改内容渲染/主题/终端时，不会互相牵连
- 扩展命令更自然：增加命令实现函数 + 更新常量即可
- 入口文件更稳定：后续迭代主要修改 `modules/`，而不是反复重写 `app.js`

### 5.3 “可发现”的终端：把导航也做成交互

V1 的 `ls` 是“告诉你有哪些东西”，但用户仍需要记住怎么输入路径。

V3 将 `ls` 输出做成可点击的“跳转命令”，核心收益是：

- 降低学习成本：新用户不需要记忆命令细节即可探索内容
- 保持一致性：点击本质仍是执行命令（不是绕开终端的另一套交互）

### 5.4 主题系统：切样式表而不是切一堆 class

V3 的主题切换采用：

- `<html data-theme="...">` 记录状态
- `<link id="themeStylesheet">` 切换 `styles.css` / `styles.light.css`
- `localStorage` 记忆用户偏好

这种方式无需引入构建工具即可保持结构清晰，且能把“主题差异”集中在 CSS 文件中。

## 6. 核心流程（V3）

### 6.1 页面启动流程

入口 [Version 3/app.js](file:///c:/Users/zcb19/Desktop/TDPB/%E5%BC%80%E5%8F%91/Version%203/app.js) 在 `DOMContentLoaded` 后按顺序：

1. 初始化评论区，暴露 `window.__comments.add(text)`
2. 从 `localStorage` 恢复主题（默认为 dark）
3. 创建内容渲染器并渲染 `DEFAULT_PAGE`
4. 创建终端（注入渲染与主题能力），启动并聚焦输入

### 6.2 `cat`：打开文章到内容区

终端 [terminal.js](file:///c:/Users/zcb19/Desktop/TDPB/%E5%BC%80%E5%8F%91/Version%203/modules/terminal.js) 的 `cat` 流程：

1. 解析输入路径（支持相对/绝对，支持省略扩展名）
2. 从虚拟文件系统解析到真实 URL
3. 调用内容渲染器 `renderPath(url)`
4. 终端输出 `opened xxx` 或 `failed to open xxx`

内容渲染器 [contentRenderer.js](file:///c:/Users/zcb19/Desktop/TDPB/%E5%BC%80%E5%8F%91/Version%203/modules/contentRenderer.js) 的 `renderPath`：

- `fetch` 获取 HTML → `DOMParser` 解析 → 提取 `<h1>` 作为标题 → 注入其余 body 到 viewport
- 渲染前后用 `switching` class 做过渡动画
- 失败时输出 `file://` 场景提示（建议使用静态服务器）

### 6.3 `theme`：切换并持久化

1. 终端命令 `theme dark|light`
2. 调用注入的 `applyTheme(theme)`
3. [theme.js](file:///c:/Users/zcb19/Desktop/TDPB/%E5%BC%80%E5%8F%91/Version%203/modules/theme.js) 更新 `data-theme`、切换样式表，并写入 `localStorage`

## 7. 已知限制与取舍

- `file://` 限制：V3 的内容加载依赖 `fetch`，在直接双击打开时可能被浏览器拦截；部署到 GitHub Pages 或本地静态服务器可解决
- 虚拟文件系统是“路由映射”而非真实文件浏览：只展示被纳入映射的内容，保持可控与可设计
- 评论仍为前端内存态：刷新丢失（符合“纯静态、零后端”的取舍）

## 8. 扩展指南（以 V3 为基准）

### 8.1 新增一个页面并可在终端打开

1. 在某目录新增静态页面，例如 `./post/xxx.html`
2. 在 [constants.js](file:///c:/Users/zcb19/Desktop/TDPB/%E5%BC%80%E5%8F%91/Version%203/modules/constants.js) 的 `FILE_SYSTEM` 中加入映射
3. 在终端执行 `ls` 找到入口，或直接 `cat /post/xxx` 打开

### 8.2 新增一个终端命令（建议做法）

1. 在 `CHEAT_SHEET` 增加命令行（保证 help 输出一致）
2. 在终端 `handleCommand` 中增加分发分支
3. 为命令写独立 `runXxx` 函数，避免 `handleCommand` 膨胀

## 9. 下一步迭代建议（面向 Version 4+）

在保持“纯静态 + 终端交互”方向下，优先级建议：

1. 内容体系从 HTML 扩展到 Markdown（文章更易写、更易维护），并引入简单模板（例如文章页统一头部/元信息）
2. 评论持久化到 `localStorage`（并提供 `clear-comments` 命令重置）
3. 虚拟文件系统支持“多文章列表”（例如 `post/` 目录下多篇文章，`ls` 可见并可点）
4. 终端输出做“最大行数”限制，避免长时间交互导致 DOM 过大
